# 项目规划书：课程作业守护者（Assignment Guardian）

## 1. 项目名称

**课程作业守护者（Assignment Guardian）** —— Linux环境下的作业全生命周期管理工具

## 2. 项目目标

大学课程通常多门并行，每门课的作业有不同的：

- 截止时间（DDL分散在各处，容易漏记）
- 提交方式（Git提交、服务器上传、平台提交）
- 格式要求（文件命名、目录结构、文档要求）

学生经常遇到以下问题：

- 忘记某个作业的截止时间，临交前才发现还没开始
- 提交时才发现文件命名不规范、少交了readme、脚本没有执行权限
- 多门课作业挤在一起，分不清哪些交了哪些没交
- 作业需求散落在PDF课件、群公告、教学平台里，每次都要翻半天
- 每门课提交路径不同，手动压缩→改名→SCP上传，重复又容易出错

**本项目开发一个面向Linux课程学生的作业管理工具，重点解决：**

- 截止时间分散，缺乏统一提醒
- 提交前低级错误频发（文件名、权限、遗漏文件）
- 作业需求信息散落各处，检索不便
- 打包上传流程重复繁琐

## 3. 项目功能

### 功能一：截止时间管家（Deadline Manager）

- 使用YAML配置文件记录各科作业的DDL、提交方式、课程名称
- crontab定时扫描，到期前分多级提醒：**7天 → 3天 → 1天 → 当天**
- 提醒方式：终端彩色高亮输出 + 生成提醒日志 + 可扩展邮件通知
- 支持命令行查询："下一周有哪些作业到期？""今天有什么要交的？"

### 功能二：作业规范自检器（Spec Checker）

提交前自动检查常见问题：
- **文件命名**是否符合配置中的命名规则
- **必交文件**是否齐全（如readme.md、源代码、报告等）
- **脚本权限**：Shell脚本是否有可执行权限（chmod +x）
- **语法检查**：Shell脚本能否通过`bash -n`语法检查
- **行尾换行**：文件是否以换行符结尾（避免Linux工具链异常）
- 生成检查报告，绿色通过/红色不通过，一目了然

### 功能三：一键打包上传（One-Click Upload）

- 读取配置文件中的提交规则（目标服务器、路径、命名格式）
- 自动打包：按课程名+日期+版本号生成规范的 `.tar.gz`
- SCP上传：自动传送到老师指定的服务器目录
- 上传后校验：对比本地与远程文件的md5值，确认上传完整
- 支持"试运行"模式（dry-run）：只展示将要执行的操作，不真正上传

### 功能四：作业需求提取器（Requirement Extractor）

- 从指定目录扫描PDF/文本文件（课件、作业说明）
- 用关键词匹配提取：**DDL日期、提交方式、命名要求、评分标准**
- 汇总到一个地方（终端输出或Markdown文件），方便快速查阅
- 支持手动添加/修正提取结果
- 不追求100%自动准确，重点是可辅助快速定位关键信息

## 4. 使用技术

项目计划使用以下Linux技术（标粗为课程要求的技术点，共覆盖**9项**）：

| 序号 | 技术 | 应用场景 |
|------|------|----------|
| 1 | **Shell脚本（bash）** | 全部功能的主控脚本 |
| 2 | **Linux系统命令** | df、du、find、stat等文件操作 |
| 3 | **grep/sed/awk 文本处理** | 日志分析、需求提取、配置解析 |
| 4 | **管道与重定向** | 命令组合、日志输出、报告生成 |
| 5 | **文件与权限管理** | 自检器检查文件权限、打包前验证 |
| 6 | **crontab定时任务** | 截止时间定时扫描与提醒 |
| 7 | **SSH/SCP远程管理** | 一键打包上传至老师服务器 |
| 8 | **Git代码管理** | 项目版本控制与开发过程记录 |
| 9 | **Python调用Linux功能** | 需求提取器PDF文本解析（备选） |

## 5. 小组分工

| 成员 | 负责模块 | 具体内容 | 重点学习 |
|------|----------|----------|----------|
| **组员A** | 截止时间管家 | YAML配置解析、DDL扫描、crontab定时任务、多级提醒逻辑、交互式查询 | Linux定时任务、Shell流程控制、文本处理 |
| **组员B** | 作业规范自检器 + 一键打包上传 | 文件命名检查、权限检查、语法检查、tar打包、SCP上传、md5校验、dry-run模式 | Linux文件系统、权限管理、远程传输、Shell函数设计 |
| **组员C** | 作业需求提取器 + 测试部署 | PDF/文本扫描、关键词提取、报告生成、功能测试、异常测试、文档编写、Git管理 | 文本处理（grep/sed/awk）、Linux调试、测试验证 |

**注意：** 组员B负责两个模块（自检器与打包上传），因为这两个模块逻辑上衔接紧密（自检通过→打包→上传），且代码量适中。组员A和组员C各负责一个独立模块加辅助工作。

## 6. 开发计划

### 第1周：开题与环境搭建
- 搭建Linux开发环境（虚拟机Ubuntu/Debian）
- 创建项目目录结构，初始化Git仓库
- 完成截止时间管家的核心功能：YAML配置读取 + DDL扫描
- 测试：扫描结果是否正确输出
- **里程碑：** 能在终端查询本周截止的作业

### 第2周：核心功能开发
- 截止时间管家：crontab定时任务 + 多级提醒 + 交互查询
- 作业规范自检器：文件命名、权限、语法检查功能
- 一键打包上传：tar打包 + SCP上传 + md5校验
- 需求提取器：基础文本扫描 + 关键词匹配
- 测试：每完成一个子功能即截图留证
- **里程碑：** 四个模块均能独立运行

### 第3周：集成与完善
- 各模块之间打通：提醒触发后可联动自检器
- 统一配置文件格式，各模块共享一份配置
- 需求提取器完善：支持PDF文本提取
- 异常处理完善：权限不足、网络断开、文件不存在
- **里程碑：** 完整作业管理流程可运行

### 第4周：测试、文档与答辩
- 全流程功能测试 + 异常测试 + 边界测试
- 修复Bug，优化用户体验（输出格式、色彩、帮助信息）
- 整理Git提交记录，确保体现阶段性开发过程
- 编写项目报告（≥10页正文）
- 制作演示视频（3-5分钟）
- 准备答辩PPT + 现场演示
- **里程碑：** 可演示的完整系统

## 7. 可能风险

| 风险 | 应对措施 |
|------|----------|
| 不同Linux发行版命令输出格式不同（如date、stat） | 优先POSIX兼容写法，测试多环境后锁定目标发行版 |
| crontab环境变量与用户Shell不同，脚本可能无法运行 | 脚本中使用绝对路径，crontab中显式设置PATH |
| SCP上传依赖网络，现场演示可能失败 | 录制操作视频+大量截图作为备用，支持dry-run模式演示 |
| PDF文本提取质量不稳定 | PDF内容提取作为辅助参考，重点依赖手动配置，降低对自动提取准确度的期待 |
| Shell脚本调试困难 | 使用`bash -x`逐行调试，关键步骤输出日志，分模块独立测试 |
| 团队成员Linux基础差异大 | 第1周集中搭建环境，老手先写好环境配置脚本给新手用，降低入门门槛 |
